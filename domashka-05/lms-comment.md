# Коментар для LMS (ДЗ 5)

Робота на macOS, тому замість `ip addr` — нативний `ifconfig`, а
`conntrack` (його на macOS немає) продемонстровано в Docker.

**1. 172.20.0.0/22**

Маска /22 = 255.255.252.0 (11111111.11111111.11111100.00000000), на хости
лишається 10 біт. Крок блоку в третьому октеті: 256 − 252 = 4.

- Адреса мережі: **172.20.0.0**
- Broadcast: **172.20.3.255**
- Діапазон хостів: **172.20.0.1 – 172.20.3.254**
- Хостів: **2^10 − 2 = 1022** (мінус адреса мережі та broadcast)

**2. Нарізка на 4 × /24** (позичаємо 2 біти, 2^2 = 4)

```
172.20.0.0/24: 172.20.0.1–172.20.0.254, bcast 172.20.0.255, 254 хости
172.20.1.0/24: 172.20.1.1–172.20.1.254, bcast 172.20.1.255, 254 хости
172.20.2.0/24: 172.20.2.1–172.20.2.254, bcast 172.20.2.255, 254 хости
172.20.3.0/24: 172.20.3.1–172.20.3.254, bcast 172.20.3.255, 254 хости
```

Перевірка: 4 × 256 = 1024 = весь /22, без дірок і перетинів.

**3. Моя адреса і підмережа**

```
$ ifconfig en0
	inet 10.20.50.89 netmask 0xffffff00 broadcast 10.20.50.255
```

0xffffff00 = 255.255.255.0 = /24. Я в підмережі **10.20.50.0/24**
(хости .1–.254), моя адреса — хост .89. Діапазон приватний (10.0.0.0/8,
RFC 1918), назовні випускає NAT; шлюз 10.20.50.1 — у тій самій підмережі.

**4. IPv6 — висновок**

```
$ ifconfig | grep inet6      # еквівалент ip -6 addr
	inet6 fe80::1445:ef77:ba10:85da%en0  (link-local)
	inet6 fd5e:875a:a143:4f74:...        (ULA від роутера)
```

Є лише link-local (fe80::) і приватна ULA (fd00::/8) — **глобальної
адреси з 2000::/3 немає, IPv6-інтернет відсутній**. Перевірки: пряме
з'єднання з IPv6-адресою Google — «No route to host»;
ipv6.test-ipv6.com не резолвиться, а ipv4-дзеркало відповідає
(мій публічний IP 91.226.253.120, ASN 16044 IPORT) — провайдер IPv6 не
видає, все йде по IPv4 через NAT; test-ipv6.com дав би 0/10.

**Опціонально: VLSM-план VPC** (з того ж 172.20.0.0/22; від найбільшої
підмережі до найменшої)

```
web 172.20.0.0/24    254 хости  (172.20.0.1–0.254)   autoscaling-tier
app 172.20.1.0/25    126 хостів (172.20.1.1–1.126)
db  172.20.1.128/26   62 хости  (172.20.1.129–1.190)
резерв: 172.20.1.192/26 + 172.20.2.0/23 (ріст / друга AZ)
```

Нарізка дає прості ACL: у db пускаємо тільки з app-підмережі, публічний
маршрут — лише у web.

**Опціонально: conntrack** (Docker, бо на macOS його немає): контейнер
172.17.0.2 качав example.com, у таблиці станів Linux-VM видно запис:

```
tcp TIME_WAIT src=172.17.0.2 dst=172.66.147.243 dport=80
              src=172.66.147.243 dst=192.168.65.3 [ASSURED]
```

Друга половина — очікувана відповідь, і dst у ній не контейнер, а
192.168.65.3 (адреса VM після MASQUERADE) — це і є запис NAT-трансляції.
Той самий механізм працює і в мене вдома, причому двічі: роутер транслює
10.20.50.0/24 у свою WAN-адресу, а публічну 91.226.253.120 навішує вже
CGNAT провайдера (у traceroute з ДЗ 4 хопи 2–3 — приватні адреси).

**Опціонально: dig AAAA vs A + curl -6**

```
dig AAAA google.com → 4 × 2a00:1450:4001:c0f::8a/::64/::65/::66 (128 біт)
dig A    google.com → 6 × 142.251.110.100–.139 (32 біти)
```

AAAA-запит успішно їде по IPv4-транспорту (DNS і зв'язність — незалежні).
Але `curl -6` підключається до ::ffff:142.251.110.100 — IPv4-mapped
адреси, тобто фактично по IPv4, а чесний конект до 2a00:... падає з
«No route to host». DNS-рівень IPv6 працює, транспортний — ні.
