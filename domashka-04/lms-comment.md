# Коментар для LMS (ДЗ 4)

Сайт: github.com. Робота на macOS, тому замість `ip route get` — нативний
еквівалент `route -n get`.

**1. dig**

```
$ dig github.com
;; ANSWER SECTION:
github.com.  60  IN  A  140.82.121.4
;; SERVER: 192.168.50.2#53(192.168.50.2)
```

Коментар: локальний DNS-сервер роутера відповів, що github.com зараз живе
на 140.82.121.4. TTL всього 60 секунд — GitHub ротує адреси в межах
140.82.121.3–.5 для балансування (нижче traceroute вже пішов на .3).

**2. route**

```
$ route -n get github.com
   route to: 140.82.121.4
    gateway: 10.20.50.1
  interface: en0
        mtu: 1500
```

Коментар: адреса не в моїй локальній мережі, тому пакети підуть через шлюз
за замовчуванням 10.20.50.1 з Wi-Fi-інтерфейсу en0. MAC шлюзу з ARP-таблиці
(`arp -a`): b0:e4:d5:43:f3:c8.

**3. traceroute**

```
$ traceroute -I -n -w 2 -m 20 github.com
 1  10.20.50.1     2.872 ms
 2  10.12.18.1     4.468 ms
 3  10.1.1.3       4.604 ms
 4  91.226.254.3   5.074 ms
 5  185.151.86.16  4.652 ms
 6  185.1.226.180  56.857 ms
 7  80.81.196.80   44.399 ms
 8  * * *
 9  * * *
10  140.82.121.3   46.064 ms
```

Коментар: 10 хопів. Перший — мій шлюз, 2–3 — приватна мережа провайдера,
з 4-го — публічний інтернет. Стрибок затримки 5 мс → 56 мс на хопі 6 — це
вихід трафіку за кордон (185.1.226.180 — peering DTEL-IX). Хопи 8–9 мовчать
(роутери GitHub не шлють ICMP TTL exceeded). Цікаве: звичайний UDP-traceroute
обривається після 7-го хопа, бо GitHub відкидає UDP-проби — до кінця
доходить лише ICMP-варіант (`-I`).

**4. tcpdump + curl (handshake)**

```
$ sudo tcpdump -i en0 -nn "tcp and net 140.82.112.0/20"
19:32:46.921664 IP 10.20.50.89.52079 > 140.82.121.3.443: Flags [SEW], seq 1977315521, length 0
19:32:46.967969 IP 140.82.121.3.443 > 10.20.50.89.52079: Flags [S.E], seq 67592, ack 1977315522, length 0
19:32:46.968157 IP 10.20.50.89.52079 > 140.82.121.3.443: Flags [.], ack 1, length 0
```

Коментар: three-way handshake. [SEW] — мій SYN (E, W — узгодження ECN),
[S.E] — SYN-ACK сервера, який підтверджує мій seq+1 (ack 1977315522),
[.] — мій фінальний ACK, з'єднання встановлене. Зайняло ~46 мс = один RTT
до GitHub, що збігається з traceroute. Наступний пакет (length 320) — вже
TLS ClientHello.

**5. Чому MAC змінюється, а IP — ні (5 речень)**

IP-адреси в заголовку пакета — це адреси кінцевих точок з'єднання (мій
комп'ютер і сервер GitHub), вони працюють на мережевому рівні (L3) і
задають «звідки й куди» на всьому шляху. MAC-адреси працюють на канальному
рівні (L2) і мають сенс лише в межах одного локального сегмента — наприклад,
між моїм ноутбуком і Wi-Fi-роутером. Кожен маршрутизатор розпаковує
Ethernet-кадр, дивиться на IP призначення і запаковує пакет у новий кадр:
source MAC стає адресою самого маршрутизатора, а destination MAC — адресою
наступного хопа (яку він дізнається через ARP). Тому на кожному з 10 хопів
пара MAC-адрес інша, а пара IP-адрес лишається незмінною від початку до
кінця. Єдиний виняток — NAT на домашньому роутері, який підміняє мій
приватний source IP на публічний, але destination IP сервера не змінюється
ніде.
